# Python 算法库平台设计

## 1. 文档目的

本文用于明确当前负责范围内的 Python 算法库平台架构。

本设计目标不是建设一个直接执行算法的计算平台，而是建设一套面向 H4 平台的 Python 算法库平台，负责：

- 标准化开发与打包
- 算法元数据注册与治理
- 算法目录查询与展示
- 算法版本和参数信息查询
- 算法发布校验与调试验证
- 后续平滑演进到私有 PyPI / 制品仓库模式

本文基于以下前提进行设计：

- 当前只聚焦 Python 算法，不覆盖 Java / Go / Shell 算法治理
- Python 算法库平台不直接与 eKuiper / Flink 交互
- Python 算法库平台主要与 H4 平台交互，向 H4 平台提供算法目录和元数据查询能力
- H4 平台负责填充参数，并将任务交给 eKuiper / Flink 执行
- 算法包分发采用 `多算法包仓库 + 私有 PyPI`

## 2. 一期建设目标

一期目标聚焦可落地能力，不追求完整目标态。

- 可开发：开发者可以按统一规范快速新增 Python 算法
- 可发布：算法包可以通过 CI 打包并发布到私有 PyPI
- 可注册：平台注册算法元数据和算法包信息
- 可查询：H4 平台可以查询算法列表、版本、参数和详情
- 可校验：平台可以校验算法元数据和算法包是否满足规范
- 可调试：平台可以提供调试执行能力用于开发联调

## 3. 方案确定

一期建议采用如下总体架构：

`多算法包仓库 + 算法 SDK + 私有 PyPI + 注册中心 + 目录服务`

其中：

- `多算法包仓库`：承载 Python 算法实现代码
- `算法 SDK`：定义统一算法契约、元数据和校验规范
- `私有 PyPI`：负责分发 wheel 包
- `注册中心`：管理算法元数据、版本和算法包信息
- `目录服务`：面向 H4 平台提供算法查询和展示接口

核心原则：

- 多算法包仓库存放源码，面向开发和维护
- 私有 PyPI 存放打包后的发布物，面向安装和分发
- 注册中心负责管理算法资产，不负责执行算法
- 目录服务负责提供查询能力，不负责运行算法
- Python 算法库平台不直接承担 eKuiper / Flink 的接入和执行

## 4. 总体架构

### 4.1 总体结构

```text
开发者
  -> 多算法包仓库
  -> CI 构建发布
      -> 私有 PyPI
      -> 注册中心

注册中心
  -> 目录服务
      -> H4 平台

H4 平台
  -> 查询算法列表、版本、参数、详情
  -> 填充运行参数
  -> 将任务提交给 eKuiper / Flink 执行
```

### 4.2 多算法包仓库与私有 PyPI 的关系

这两个部分需要分开设计，原因不是“目录形式不同”，而是“职责不同”。

- `多算法包仓库`：存放算法源码、测试用例和打包配置，面向开发过程
- `私有 PyPI`：存放打包后的 wheel，面向发布和安装过程

可以简单理解为：

- 多算法包仓库是开发态资产
- 私有 PyPI 是运行态制品

如果两者不分离，后续在“开发、发布、安装、回滚”几个环节上容易混淆职责。

### 4.3 数据流

一期建议采用如下数据流：

1. 开发者在多算法包仓库中开发新的 Python 算法
2. CI 对算法包进行测试、校验和打包
3. 打包产物发布到私有 PyPI
4. 发布工具或 CI 调用注册中心，登记算法元数据和算法包信息
5. 目录服务对外提供算法目录和详情查询接口
6. H4 平台查询算法、版本、参数和说明
7. H4 平台填充参数后，将任务提交给 eKuiper / Flink 执行

这里需要明确：

- Python 算法库平台只负责“管理和查询”
- H4 平台负责“使用这些信息组织任务”
- eKuiper / Flink 负责“最终执行”

## 5. 服务边界

### 5.1 算法 SDK

负责定义统一的 Python 算法开发规范：

- `BaseAlgorithm` 或统一接口基类
- 算法元数据定义
- 参数、输入、输出结构定义
- 入口函数约定
- 校验器
- 本地调试工具

SDK 是算法包和平台规范之间的桥梁。

### 5.2 多算法包仓库

负责承载多个独立 Python 算法包源码：

- 每个算法包独立打包
- 每个算法包独立测试
- 统一依赖 SDK
- 统一由 CI 打包并发布到私有 PyPI

### 5.3 私有 PyPI

负责：

- 保存和分发 wheel 包
- 按包名和版本提供下载
- 作为算法包安装来源

不负责：

- 算法元数据管理
- 前端展示
- 版本状态治理
- 算法执行

### 5.4 注册中心

注册中心是算法资产的权威写入口，负责：

- 算法注册
- 算法版本登记
- 算法元数据维护
- 算法包信息绑定
- 元数据校验结果记录
- 发布状态记录

注册中心回答的问题是：

- 当前平台里有哪些算法
- 某个算法有哪些版本
- 某个版本对应哪个算法包
- 某个版本当前是否可被查询和使用

### 5.5 目录服务

目录服务是算法资产的读入口，面向 H4 平台提供：

- 算法列表查询
- 算法详情查询
- 算法版本查询
- 输入输出和参数查询
- 标签、分类、描述等展示信息

目录服务不负责管理状态，只负责提供稳定查询接口。

## 6. 核心模型

一期建议只保留两个核心模型，避免过度建模。

### 6.1 算法定义

算法定义就是平台内的核心算法元数据。

建议字段：

- `code`：唯一标识
- `name`：名称
- `version`：版本
- `category`：分类
- `description`：描述
- `inputs`：输入定义
- `outputs`：输出定义
- `params`：参数定义
- `entrypoint`：运行入口
- `resources`：资源需求，例如 CPU / 内存 / 超时
- `requirements`：依赖信息
- `tags`：算法标签，例如清洗、转换、特征工程、规则判断
- `status`：当前状态

说明：

- 这是平台查询和展示时最核心的一份数据
- 它既包含算法本身的业务信息，也包含 H4 平台最关注的参数和输入输出信息

### 6.2 算法包信息

算法包信息用于描述某个算法版本对应的发布物位置。

建议字段：

- `code`
- `version`
- `package_name`
- `package_version`
- `repository_url`
- `artifact_type`
- `filename`
- `sha256`

说明：

- `算法定义` 负责业务元数据
- `算法包信息` 负责制品定位
- 二者通过 `code + version` 关联

### 6.3 可选的注册记录

如果一期需要保留注册和校验过程信息，可以增加辅助记录：

- `code`
- `version`
- `register_status`
- `validate_status`
- `message`
- `created_at`

这一层不是必须，一期可以按实现复杂度决定是否保留。

## 7. API 设计

一期建议保留以下核心接口，并给每个接口明确功能说明。

### 7.1 `GET /algorithms`

功能说明：

- 查询算法列表
- 支持按名称、分类、标签、状态筛选
- 供 H4 平台展示算法目录

主要返回：

- 算法基础信息列表
- 分类、标签、状态等展示字段

使用场景：

- H4 平台算法目录页
- 算法筛选和搜索

### 7.2 `GET /algorithms/{code}`

功能说明：

- 查询指定算法的基本信息和当前版本概览

主要返回：

- 算法名称、分类、描述、标签
- 当前可见版本或默认展示版本

使用场景：

- H4 平台查看算法详情

### 7.3 `GET /algorithms/{code}/versions`

功能说明：

- 查询指定算法的所有版本信息
- 供 H4 平台选择使用版本

主要返回：

- 版本号
- 状态
- 发布时间或登记时间

使用场景：

- 算法版本选择
- 历史版本查看

### 7.4 `GET /algorithms/{code}/versions/{version}`

功能说明：

- 查询指定算法版本的完整元数据

主要返回：

- 输入定义
- 输出定义
- 参数定义
- 资源需求
- 依赖信息
- 算法包信息

使用场景：

- H4 平台配置参数表单
- 查看版本详情

### 7.5 `POST /algorithms/register`

功能说明：

- 注册新的算法或新的算法版本
- 写入算法元数据和算法包信息

主要入参：

- 算法定义
- 算法包信息

主要返回：

- 注册结果
- 失败原因或校验提示

使用场景：

- 发布流水线自动登记
- 运维或开发人员手动登记

### 7.6 `POST /algorithms/validate`

功能说明：

- 校验算法包和元数据是否满足平台规范

主要校验内容：

- 元数据字段完整性
- 包地址可用性
- 入口定义是否合法
- 参数结构是否合规

主要返回：

- 校验是否通过
- 详细错误信息

使用场景：

- 注册前校验
- CI 发布前校验

### 7.7 `POST /algorithms/{code}/execute`

功能说明：

- 对指定算法进行一次调试执行
- 仅用于开发调试或联调，不作为正式生产执行接口

主要入参：

- 测试输入
- 测试参数

主要返回：

- 调试执行结果
- 执行日志或错误信息

使用场景：

- 开发调试
- H4 平台联调验证

## 8. 发布与注册流程

### 8.1 开发与打包

开发者基于 SDK 创建算法包，补充：

- 算法元数据
- 入口函数
- 参数定义
- 输入输出定义
- 依赖信息

CI 执行单测、结构校验和打包。

### 8.2 发布到私有 PyPI

CI 将 wheel 发布到私有 PyPI，并记录：

- `package_name`
- `package_version`
- `repository_url`
- `sha256`

到这一步说明：

- 算法包已经发布成功
- 但还没有进入平台可查询状态

### 8.3 注册中心登记

CI 或发布工具调用注册中心登记：

- 算法定义
- 算法包信息
- 版本信息

注册中心完成：

- 唯一性校验
- 元数据完整性校验
- 包信息绑定
- 入口和参数结构校验

### 8.4 目录服务展示

目录服务读取注册后的算法信息，对 H4 平台提供查询能力。

### 8.5 H4 平台使用

H4 平台通过目录接口查询：

- 算法列表
- 算法版本
- 参数定义
- 输入输出说明

随后由 H4 平台负责参数填充和执行编排。

## 9. 代码仓结构

一期建议采用单仓多模块结构：

```text
algorithm_repository/
  docs/
    plans/
  sdk/
    algorithm_sdk/
  algorithms/
    algo_missing_value/
    algo_xxx/
  services/
    library_platform/
      registry/
      catalog/
      debug/
  tools/
    scaffold/
    publish_cli/
    validate_cli/
```

说明：

- `sdk/`：统一算法契约
- `algorithms/`：多算法包源码
- `services/library_platform/registry/`：注册相关能力
- `services/library_platform/catalog/`：目录查询相关能力
- `services/library_platform/debug/`：调试执行相关能力
- `tools/`：脚手架、发布和校验工具

一期建议优先做成一个后端项目中的多个模块，而不是一开始拆成多个独立微服务。

## 10. 后续扩展

以下能力建议放到后续阶段，不纳入一期范围：

- 审批流
- 更完整的状态机
- 更复杂的发布治理
- 多语言算法扩展
- 更复杂的执行引擎对接
- 多环境和多租户治理

## 11. 结论

本设计建议将当前项目定位为：

**面向 H4 平台的 Python 算法库平台。**

它负责标准化开发、算法元数据注册、目录展示、版本查询、校验和调试能力，但不直接承担算法执行。

在该定位下：

- 多算法包仓库是源码层
- 私有 PyPI 是发布层
- 注册中心是管理层
- 目录服务是查询层
- H4 平台是使用方

这几层协同后，才能支撑一期“可开发、可发布、可注册、可查询、可调试”的 Python 算法库平台。
