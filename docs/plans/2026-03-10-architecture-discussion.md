# 平台基座与算法库系统架构

## 二、 架构模式选择探讨
在“控制面（基座）与数据执行面（算法库）分离”的指导思想下，主要有以下两种可行路线：

### 方案 A：原生微服务调度架构（推荐）
*   **架构形态**：算法库系统作为一个纯粹的微服务网关，依托成熟的非阻塞异步任务队列服务（如 Celery + Redis）进行任务下发，工作节点（Worker）并发处理计算。
*   **优点**：
    *   **开发难度低、落地快**：与目前主流的 Python 微服务技术栈（如 FastAPI）完美契合。
    *   **针对性强**：Celery 自身天然支持 DAG（有向无环图）的串行、并行及拓扑控制逻辑。
*   **缺点**：若不同算法模块之间存在严重互斥的环境依赖（如核心 C 库版本冲突），则在同一个节点内管理算法运行环境存在一定挑战。

### 方案 B：云原生容器化调度架构
*   **架构形态**：系统不长期运行用于计算的 Worker 进程。每次执行任务，均将用户的流程图翻译为 Kubernetes 工作流规范，由 K8s 动态创建 Pod（容器）进行计算，算完即毁。
*   **优点**：最高级别的资源与环境物理隔离，算力调度极其灵活。
*   **缺点**：技术门槛与服务器资源要求极高；每次计算都有启动容器的滞后时间，不适合高频的小规模串行任务。

=> **研讨结论与基础建议**：由于方案 A 更注重效率且能应对绝大多数数据处理场景，推荐作为平台一期的落地路线。

## 三、 功能职责与数据边界
为确保解耦后两个阶段互不干涉、“各司其职”，我们明确以下边界：

### 1. 平台基座（Base Platform）—— 控制枢纽
*   **定位**：负责交互、审批、流程编排及控制链路闭环。
*   **功能边界**：
    *   管理用户权限与项目归属。
    *   提供可视化的连线画板，并将用户绘制的图结构（DAG）持久化到基座的物理数据库中。
    *   负责下发完整的任务链条指令，以及实时接收进度将状态反映到页面图表节点上。
*   **不可跨越的点**：绝对不参与算法的内部执行逻辑，不需要预装任何算法相关的底层依赖包。

### 2. 算法库系统（Algorithm Repository）—— 执行引擎
*   **定位**：负责接收命令、分配算力、执行数据处理逻辑并保存产物。
*   **功能边界**：
    *   管理庞大的算法资产（定义输入什么、输出什么参数规范）。
    *   接收基座发来的指令后，按依赖先后顺序调度机器跑对应算法。
*   **不可跨越的点**：不关心当前任务属于哪个前端具体用户（无 Session 控制概念），纯纯的“拿到指令就干活的打工机”。

### 3. 数据隔离边界
*   **数据库**：基座的业务库 与 算法系统的任务记录库 绝对分开，禁止任何形式的跨表跨库 Join。
*   **业务流大文件传输**：两个系统之间**坚决不通过接口直接传输 CSV/Parquet 等实质性宽表数据**。所有源文件或步骤中间产物，统一存放于 **MinIO 对象存储** 之中。两个系统交流时，只交换一个指针字符串（即类似于 `s3://xxx` 的文件的 URI 路径）。

## 四、 核心接口交互设计方案指引
为了将上述功能运转起来，两系统的接口交互逻辑缩减为极简的三个主要动作：

1. **“有什么能力？” (同步暴露)**
   *   动作：基座在打开页面时，询问算法系统“你有多少种算法组件？”
   *   机制：算法系统吐出一个极其规范的文档列表（包含所有必需的输入框类型与校验条件），基座据此在页面左侧渲染出算法图标及参数侧边栏。
2. **“干活出发！” (异步下发)**
   *   动作：用户在基座点击【执行】按钮。
   *   机制：基座将这张庞大的逻辑图组装成一份只包含参数和 MinIO 下载地址的“任务书”，交给算法系统。算法系统收到后立刻给个“已接单”回执跳过等待，随即转入后台悄悄排队计算。
3. **“事办完了！” (事件回传)**
   *   动作：算法系统里某个算法运行结束并把结果放回 MinIO。
   *   机制：算法系统并不等基座来问，而是主动朝**消息队列中间件（MQ）**大喊一声“某某环节刚才由于发生了异常而停止”或者“某某节点执行成功，结果放在了这个 URI”，基座系统倾听并将其转换至可视化前端。
