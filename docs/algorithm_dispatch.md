# 算法调度引擎 (Orchestrator) 架构设计讨论

## 一、 算法设计与调度引擎的逻辑关系

在我们的系统中，**“算法组件”**和**“调度引擎”**是相互独立又紧密咬合的两个齿轮。

### 1. 算法组件的设计：颗粒度极细，万物皆平级
为了给产品和用户提供最大化的灵活性，我们将所有的算法组件（算子）颗粒度设计得非常小。
无论是极度轻量的“清除缺失值”，还是重量级的“深度学习模型训练”，它们在代码层面都继承自同一个**基类 (`BaseAlgorithm`)**，地位完全平等。大家都可以被用户在前端画布上自由拖拽和连线。

### 2. 调度引擎的职责：从大JSON到真实分布式任务
当用户连好线点击“运行”时，前端会下发一个庞大的 JSON 描述文件（Flow DSL）。
调度引擎（Orchestrator）的任务就是接收这图纸，解析出执行的先后顺序，并把它们派发给后端的计算集群（如 Celery Worker）去执行。如果节点互相依赖，比如 B 需要 A 的输出，引擎还要负责把 A 算出的数据下载链接传给 B。

---

## 二、 核心破局点：为何要做“智能融合调度”？

### 传统调度模式（“搬砖困境”）
在细颗粒度的算子下，如果完全依赖传统的分布式调度：
每次执行一个节点，Celery Worker 都需要：`去 MinIO 下载庞大的数据集文件 -> 放入内存计算 -> 计算完毕写回 MinIO`。
如果用户连了三个连续的轻量级清洗算子（比如：清除 -> 标准化 -> 过滤），底层的执行路径将是：

```mermaid
graph LR
    subgraph 传统调度：严重依赖网络 IO
        A[MinIO对象存储] -->|下载文件| B[Celery: 缺失值清理]
        B -->|上传文件| A
        A -->|下载文件| C[Celery: 标准化]
        C -->|上传文件| A
        A -->|下载文件| D[Celery: 数据过滤]
        D -->|上传文件| E[最终结果]
    end
```
**痛点**：对于轻量级算子，这种反覆读写磁盘和网络的 IO 开销，是纯粹在内存里计算耗时的数倍甚至几十倍。如果用户连线多了，系统会被极其低效的搬运拖垮。

### 我们的创新点：智能融合（Smart Bundling）
针对上述痛点，我们设计了带有**图层分析优化的 Orchestrator**：
我们在算法元数据中引入了一个特殊的标识 `execution_mode: in_memory`（纯内存清洗类算子）。
当引擎解析图纸时，如果发现连续出现几个没有其他分支干扰的 `in_memory` 算子，调度引擎会在底层**动态地将它们合并压缩成一个巨大的“级联计算任务”**下发。

```mermaid
graph LR
    subgraph 我们的创新：图融合内存击穿
        X[MinIO引擎仓库] -->|全层仅下载 1 次| Y((组合工作器 Celery Bundle Task))
        
        subgraph 组合工作器内存区域
            Z1[动作1: 缺失值清理] -->|DataFrame 内存引用| Z2[动作2: 标准化]
            Z2 -->|DataFrame 内存引用| Z3[动作3: 数据过滤]
        end
        Y --> Z1
        Z3 -->|全层仅上传 1 次| W[最终结果落盘]
    end
```
这就像把流水线的搬运工取消了，由一个熟练工在一个工位上把三套工序一次性做完再入库。大幅压缩了不必要的 IO 通信。

## 三、 本套架构带来的核心优势

1. **破局性能瓶颈，节省海量通信成本**：
   合并计算模式直接干掉了 60% - 80% 的 MinIO 文件拉取与分布式任务派发队列的开销。对于大体积时序数据处理，执行效率呈现指数级上扬。
   
2. **前后端解耦，前端享有无限自由**：
   有了底层的智能优化托底，业务侧和产品经理再也不用担心“组件如果拆得太细，服务器会不会卡死”。前端可以随心所欲暴露最细粒度的算子给用户拖拽，“所见即所得”，后端完全能够自适应承接。

3. **对用户透明的细粒度平滑反馈**：
   虽然后端把几个算子放在一个工作器里跑，但在那个单一的合并循环里，每跑完一个微小的步骤，后端仍会发出状态更新脉冲。这样在前端界面的大屏幕长画布上，每个小算子节点依旧能平滑完美地依次亮起“就绪”、“成功”的绿灯。

4. **预检隔离，保障平台高可用**：
   在真正唤醒分布式资源进行上述运算前，调度器设有独立的前置关卡，直接识别死循环和非法参数连线，在未耗损任何实质性算力时予以拦截，严防单体错误耗光集群池。

## 四、 潜在风险点与演进计划（注意事项）

这套基于内存击穿的高级架构对开发和运维提出了一些要求，我们需要注意以下风险：

1. **OOM（Out Of Memory）内存溢出风险：**
   将连续算子都捆绑在一起计算，意味着一份巨大的数据集可能在内存中驻留或引发拷贝放大。如果一次涌入数百 GB 的数据流水，可能击穿 Celery Worker 的内存上限。
   * **应对方案**：在未来的优化中，对单个 Bundle 施加阈值限制（例如最多限制串联 5 个算子），或者评估采用 Pandas 的 `chunking`（分块流式读取计算）替代全量读取。

2. **分支依赖复杂度：**
   我们的智能打包逻辑目前仅适用于“严格一对一”（线性链式的 `in_memory` 节点）。如果有节点既输出给下一个内存算子，又作为分支输出了一个长效模型训练任务（触发了并行支点），必须强制斩断打包并在该节点将结果落地（Checkpoint）。否则数据无法复用。我们在底层代码中已加入这一策略的分界逻辑，需长期严格遵守。

3. **断点恢复无法精确到捆绑内：**
   因为多个节点合并成为了底层的一个大 Celery 任务，如果执行到中途挂掉了，系统级别的“断点重试”机制只能从捆绑包的起点重新开始算，而无法在内存的中间态直接接续。由于在 `in_memory` 算子场景下计算非常极速，从链条起点重试的开销是完全可以接受的。

---

> **阶段结论与建议**：
> 这一版基于图论与内存下钻相融合的 Orchestrator 设计，用“打包思想”化解了“拖拽式算子”长期以来的重型 IO 痛点。不仅大大加宽了性能上限，也为开发算子铺垫了傻瓜式的扩展之路。强烈建议围绕该基础代码实施下一步落地和联调验证。
